# Project Rules

## Board Implementation
- **GPU, LCD, Touch panel, DMA, INTC, RTC, Main battery** - These peripherals must be always implemented at first hit to prevent debugging for 10 hours and finding out that we just stubbed them. Fully. End-to-end. No exceptions. At all. **Audio** might be a careful stub - bad audio stub almost always causes the deadlock somewhere at OS level that renders same 10 hours debugging. Do not implement backup battery.
- **UART** and other serial/debugging **TX** must be implemented on first hit.
- **All data is only in references** - Load or find data in references/ directory. If there is no data/sheets needed for your board/SoC go and download online. Put there. PDFs, Linux sources, QEMU.
- **NEVER READ PDF AS TEXT** - SoC sheets often have complex graphics exploded via text export of PDF. Parse PDF into text only to find pages. Then take the page, export into picture, and read the pictures.
- **Kernel has parked into scheduler idle, nothing happens** - you are either creating a new board, or debugging new ROM. This means that RunApps didn't fire - filesys.exe broke somewhere, device.exe broke somewhere or gwes.exe. Bisect their entry points. Bisect CreateThread, CreateProcess from kernel. Once you found this, write it down at least somewere to never rediscover. This happened many many times, you would spend 40% of session to rediscover that CPU is busy idling and having nothing to do. A written reminder in tasks list, checklist, user memory, trace hook probe that says "this address is kernel scheduler idling" would kill potential money waste. Hook everything you need nuclearly: hook LoadLibrary, CreateProcess,CreateWindow, CreateThread, ShowWindow. Read strings, flags. Always nuclear.
- Any prexisting failure path code, e.g. accessing an incomplete/absent peripheral, always FATALs CERF (so if you see such access as last log line, this never means that CERF passed garbage into guest and it corrupted, this means CERF exited)
- **Always implement debug TX / NKDbgPrintfW / OEMWriteDebugString path**. Don't wait for UART unimplemented peripheral error - not all devices implement OEM debug strings through and in fact they might not implement any AT ALL. Once you see banner like "Windows CE Kernel for ARM (Thumb Enabled) Built on Jan 23 2006" in decompilation, figure out where it goes and implement the corresponding mechanism. If it goes nowhere (and that happens OFTEN! - path is nulled by OEM) - hook the nulled sink with a CRC-gated file under ``cerf/tracing/<bundle>/nkdbg/`` (production-built; see subsystems.md § Kernel debug output). The OEM Debug String func produces absolutely critical data for any debugging - kernel exceptions, OEM halt messages, tons of useful stuff - so avoiding this path is forbidden. All guest debug output routes through the ``KernelDebugSink`` service: a live UART/serial peripheral's TX and a hooked nulled-OEM sink alike call it, and it emits each line to the ``Nkdbg`` log channel (``[NKDBG]``) and the ``HwScreen`` debug console. Never open-code ``LOG`` + ``HwScreen::AddLine`` in a peripheral - route through the sink.
- **Always implement SoC reset/power down/suspend paths** - some devices utilize reset during boot legally. Some devices have power down or use power down mode as unrecoverable crash state. We dont fully implement suspend in any proper way for today - the SoC should go back to regular state instantly. However the power down and reset paths must be fully implemented. When reset/powerdown initiated, use shared ``GuestPowerNotifier`` service to notify UI. CERF-initiated resets route through the shared ``GuestCpuReset`` service so the SoC latches a true reset cause and reset-line peripherals reset; a causeless reset sends cause-checking guest boot paths into sleep-resume hangs.
- **User-facing board quirks live in the launcher boards list** - a note is for a board-SPECIFIC deviation or limitation a user must know and could NOT guess from the running UI (input/keypad/keyboard mapping, unsupported or broken peripherals, resolution caps, guest-additions caveats, boot quirks). It belongs in `launcher/supported_devices.py`'s per-board ``notes``, which surface in the launcher side panel and extend a ROM's own ``cerf.json`` notes. A note is NEVER for a standard feature that boards normally have and that is already self-evident in the UI - a working battery indicator, a status-bar widget with its own tooltips, a mouse cursor. If the would-be note reduces to "this board has \<a normal, visible feature\>", it is not a note. When you hit a genuine board-specific quirk, propose adding it to the user rather than writing it unprompted.

## Host UI

- **No font-glyph icon rendering** - host widget / status-bar icons are drawn with GDI primitives (`FillRect`, `Polygon`, `Rectangle`, `Arc`), never as text glyphs from a symbol font (`Segoe MDL2 Assets`, `Segoe UI Symbol`, `Wingdings`, emoji). Symbol fonts are not present on every supported host (pre-Win10 Windows, Wine), and a missing face never fails - GDI silently substitutes a font without those codepoints and the icon renders as a `.notdef` box.

## JIT, MMU, CPU, Peripherals, Boards Code Changes Rules

The enumerated modules are the most fragile on CERF ordered by the priority. E.g. JIT is the place where you change code the MOST careful way. No mistakes in JIT can be done. Prefferable this spreads accross other modules, tho JIT is the worst. The harm in JIT is silent and causes weeks of debugging and tons of money spent on debugging. Same goes for shitcode/destruction in MMU, CPU, Peripherals, SoC configuration, - basically the entire emulation core.

Once you need to update something in any of these modules, the correct workflow is always:
- You have or download official sheets into ``references/`` directory
- You find the sections regarding what you are going to change and CONFIRM that official document is aligned with your conception
- You do the code change
- You run ``/verify`` and ask the subagent to validate your implementation against common sense and sheets you downloaded into `references/``
- Final verdict decides everything

Every new code and change JIT/MMU must be done through this workflow. Every SoC/peripheral setup must go through this (or decompilation/active debugging through trace manager/etc - any source of REAL information is GOOD)

**Remember**: Your training data forces you to generate paragraphs and citations in code. You probably will skip this section often and that would be a mistake every time. Once you re-verify your implementation+citation based on you made based training data only against the official SoC/ARM/etc. document, you would be surprised that all paragraphs, encondings, bits, addresses, timings YOU MADE UP and they are COMPLETELY INCORRECT compared to what official document shows! Think how much harm random data you made up can cause and how long debugging could be just to figure out that you used random numbers somewhere instead of real document data available for $0 in ``references/``.

**Hot path performance** is what we NEVER degrade, it is ALWAYS a concern and it is a subject to violate code style rules (of course after a discussion if a compromise really needed). If you see a pre-existing code in JIT that obviously has performance harmed, tell user that this part could be optimized later (or now). New is always as optimized as possible, no violations here allowed.

**An instruction is ALWAYS implemented fully**. No half-implemented instructions are allowed. If an agent implements an instruction that handles one specific case while it should handle much more cases, it is a bailout and heavy rules violation. The fatal'ing instructions are non-runtime for today ones aligned with user agreement. If user never told "OK, you can skip this instruction implementation", then you can't skip implementation for this instruction. There has to be a serious reason to skip instruction implementation. Once there is a task to implement X instructions / instruction Y,X,Z - you are implementing them fully. Unfinished new implementation are reverted and treated as rules violation - those are damage to code, damage to user's wallet. UND/FATAL'ing stubs in new instructions are extreme exception agreed with user on concerete facts and terms. The reviewer agent should be extra carefully about this.

**Carefully choose UND or CerfFatalExit for stubbed JIT paths** - don't put UND on unimplemented paths in JIT. If a path might receive a runtime call, this must be visible - CerfFatalExit. UND is rather a very special case, never a solution for stub. UND makes execution go further and starts the extreme corruption chain. UND instead of CerfFatalExit already caused dozens hours of debugging. Don't violate the loud stub rule! If noticed that we use UND somewhere where it clearly should be FATAL - tell user, do not wait or postpone. This is not a case for discussion if you are proposing user bail out options. Read previous point. Everything is always fully implemened. If you are doing damage, then you are doing damage with whatever framing you use.

**There are ZERO known/pre-existing bugs in JIT/MMU/etc** - if you found a bug that you didn't discuss with user - stop and give user data. This is extremely important. This rule is also spreading on the rest of code, not only JIT/MMU.

Heuristics and similar ambigous approaches are forbidden in JIT/MMU. They are forbidden at any level where problem is obviously can't be handled at 100%, consider such heuristic approach an architecture damage. It's fine e.g. in board detector, where we just scan ROM to find something that points at specific board. But at JIT level that would be clear architecture damage.

## Speed Is The Enemy
- **Slow is correct. Fast is wrong.** - You are trained to produce output quickly, cover ground, look productive. That training is your enemy here. Every time you feel the urge to write code quickly, that urge is the exact moment you are about to guess. Resist it. The user is not paying for volume of output. The user is paying for correctness. One correct function per hour beats ten guessed functions per minute. If you feel yourself moving fast, you are moving wrong.
- **Before writing any peripheral register handler, the chip datasheet entry (or BSP source line) for that register must be visible in the conversation above.** If you have not pasted the reference in this session, you have not verified. "I already know what it does" / "it's a simple status register" / "the default is probably X" all mean you are about to guess. Open the datasheet. Read the entry. Then write the code.

## Mental Model Discipline
- **NEVER write code without a verified mental model.** Before touching code, state your understanding as a testable claim. Then verify it - by reading the relevant reference (chip datasheet, BSP source, ARM architecture reference manual), or a log line from a diagnostic you ran in THIS session. If confirmed, proceed. If not confirmed, you have not verified. Do not code. "I think I know" is not verified. "It's probably X" is not verified.
- **Declare success at the scope the user cares about, not the scope easiest to instrument.** An internal progress metric - JIT block translation count rising, memory plateauing, no watchdog fire, a specific subsystem's mutex releasing - is evidence that the measured layer isn't deadlocked. It is NOT evidence that the emulator works. Acceptance criteria for "fixed" must match the user's observable criteria (window paints, `connect()` returns, DHCPACK on the wire, shell responds to clicks). When the metric scope and the acceptance scope diverge, the metric proves nothing the user asked about, and calling it "fixed" is fabrication of a success that was never measured.
- **Symptom shape constrains the hypothesis** - a defect's location, geometric shape, color, and timing are themselves evidence; a candidate mechanism whose output doesn't paint those specific pixels in that specific way is wrong regardless of how broken the mechanism is in isolation.

## Investigation Scope Discipline
- **Architectural claims need the same verification as code claims.** "X is never called", "Y is never set", "Z bypasses W" are factual claims, not observations. Each one requires a grep + READ of the call sites cited as proof - not a guess about what the code "probably" does. Writing an architectural failure cascade ("X never runs → Y leaks → Z is missing → whole subsystem needs rework") without that verification is fabrication dressed as analysis. The user reads it as fact, plans against it, and every minute of their review is wasted before the foundational claim is disproved.
- **Crash stacks are ground truth - reconcile every "doesn't run" claim against them.** Every frame in a crash stack is a function that WAS running. If frame [N] shows `Foo+0xABC`, you may not later claim "Foo isn't installed here" or "Foo's setup didn't happen" without first pointing to the exact frame and explaining the contradiction. The crash stack overrides any theory. If your theory contradicts the stack, the theory is wrong - not the stack.
- **Match the reference; don't evaluate alternatives** - when CERF has a concrete reference design (Microsoft sample source under `references/`, an existing committed subsystem, a documented prior implementation), the engineering question is "what does the reference do?" rather than "what are the trade-offs of alternative approaches?", and reaching for trade-off framing when a reference settles the question wastes cycles and produces confabulated reasoning.
- **Narrow the path before globbing** - the tree is large enough that whole-tree Glob patterns (`references/**/...`) time out; restrict to a specific BSP / SDK subdirectory to get results in time. NEVER DO GLOB IN REFERENCES/ DIRECTORY! ALWAYS DO ``ls`` FIRST - SEE WHAT DIRECTORIES IT HAS. Most likely you will find what you are looking for instantly.
- **Any CERF forked submodule is first-class CERF code, not external/untouchable** - when an investigation converges on a forked submodule's runtime behavior, check the fork's git log for our own patches before assuming vanilla behavior, because bugs surfacing through a fork are by default OUR bugs in OUR patches, and extending or fixing the fork is valid work in the same way patching any other CERF directory is.
- **Force-clean a forked submodule's build artifacts before debugging fork-related symptoms** - incremental build systems can leave stale `.obj` files in a fork's build tree even when source / object timestamps look consistent, so before concluding that fork runtime matches what its source says, delete the matching object plus the linked library and rebuild from scratch.

## WinCE Accuracy
- **CERF is the virtual platform - every CE binary runs intact; nothing CE is reimplemented host-side** - `nk.exe`, `coredll.dll`, `gwes.exe`, `filesys.exe`, `device.exe`, COM / OLE, LPC, the full TCP/IP stack, `commctrl`, `aygshell`, `ceshell`, `shell32`, every userspace subsystem and every kernel-mode binary run as the ROM's own ARM code unmodified. When something misbehaves, the fix is in the virtual-platform surface CERF presents (memory map, peripherals, interrupts, OAL surface, emulated drivers), never in a host-C++ rewrite of the offending CE binary. Host C++ in CERF is bounded by virtual-hardware emulation + emulated peripheral drivers; it does not extend into CE userspace or CE kernel reimplementation. Any proposal that reads "let's mock / thunk / replace / reimplement <some CE binary> in host C++" is rejected at framing.
- **Numeric conversions go through a tool, never mental arithmetic** - every decimal↔hex conversion, signed↔unsigned reinterpretation, bit-mask decode, and hex-arithmetic step at ≥16 bits goes through a Bash / PowerShell / Python / `printf` tool call. Hand-done conversions at 32-bit scale silently produce wrong answers that read as authoritative - the wrong value usually lands on the same "shape" as the right one (leading bytes match, rough magnitude matches) so the error looks like a plausible constant, and a downstream investigator then builds a whole theory on top ("the tool must be lying", "the chip must swap these bits", "this field must mean something else") before anyone questions the original arithmetic. This cascades directly into wrong code being shipped. If you catch yourself thinking "I know hex, `-N` is obviously `0xX`" / "saves a step to just type the answer" / "the tool must be wrong, let me check the disasm" - those are the cascade's start signals; run the tool. The `calc` skill is the forcing function - invoke it when you catch the urge. If a tool result contradicts a value you already wrote, retract the previous value in plain language rather than rationalizing the difference.
- **No hacks** - emulated peripheral behavior comes from chip datasheet + matching BSP source, not from invented values. No hardcoded screen dimensions - screen size comes from kernel signaling the resolution into LCD peripheral.
- **Stubs must fail honestly** - return failure, not fake success. Log the function name and args. Nobody is going to debug 10 hours to realize that we lack some logic. (No, you can't "defer" work and reference this rule - once feel an urge to do so, just stop.)
- **No Sleep hacks** - `Sleep()` / busy-wait loops in emulator code are guilty until proven by chip datasheet timing requirement or BSP source reference. Use proper signaling, not delays.
- **Corruption = halt** - if the emulator detects invalid PC, unmapped memory, corrupted kernel structures: abort immediately. Never log-and-continue on corrupted state.
- **External-format compatibility cannot be tested by round-tripping your own writer through your own reader** - when implementing a file format, wire protocol, or any binary structure that's supposed to interoperate with an external system (real OS, real binary, real device), validation requires bytes produced by that external system going through your reader, OR bytes produced by your writer being consumed by the external system. Round-tripping your own output through your own code proves nothing - it only verifies your reader and writer agree on whatever they agree on, including hallucinated constants. If the format is "supposed to be compatible with X", you have not tested compatibility until X has touched the bytes.
- **Per-device facts come from the ROM - never from `cerf.json`/`meta`, RAM heuristics, or image hashes** - every per-device operational decision (board, SoC, memory map, OS version, and per-ROM / per-CE-version variation *within one board*) is derived from the ROM's own content, the way `BoardDetector` fingerprints a board and `PageTableBuilder` reads the OAT. The question "where do I get fact X for this device?" always answers "read it from the ROM." Three sources are forbidden for this by definition, each with its own failure shape:
  - **`cerf.json` / `meta`** - `cerf.json` is OPTIONAL and has no value for `cerf.exe`; `meta` is launcher-authored display strings written from a remote manifest. Gating any behavior on it breaks the documented bare-`cerf.exe` path the instant the file is absent or the manifest is wrong - the device silently misbehaves (e.g. won't render) with no error. It is a honeypot: `meta` carries named fields (`os.ver_major`, `soc_family`) that look exactly like the operational fact you want, so the pull is strong - resist it; the answer is in the ROM, not the sidecar.
  - **Runtime RAM / coprocessor heuristics** (reading guest memory, page tables, cp13/FCSE state, or kernel structs to infer a fact) - forbidden because: (a) it guesses a fact that is deterministic and known pre-boot; (b) the signal usually cannot even distinguish the cases you need (FCSE-vs-ASID is the CE≤5 vs CE6+ split, so it cannot tell two FCSE-era versions apart); (c) ordering - board/service setup runs before the kernel establishes that state, so there is nothing to read at decision time; (d) FCSE low-VA guest reads are themselves unreliable; (e) it couples a static, pre-known fact to volatile, process-dependent runtime state.
  - **Whole-image CRC32 / hash** - matches ONE exact image, never a class; a board has many ROMs (revisions, regions, generations, future user dumps), so it needs an unbounded checksum list and any unseen ROM gets no behavior. It is the anti-fingerprint - generalizes to nothing - and fails silently on any byte change.
  - **The test:** would this still work on a bare ROM with no `cerf.json`, on a ROM revision you have never seen, decided before the kernel boots? If not, you are using a forbidden source.

## Service Locator & Architecture
- **Hot path performance achievement** is the reason to violate code style rules. If it 
- **No service locator bypass** - never cache service pointers in statics, globals, or init-time copies. Services accessed via `emu.Get<T>()` through captured references (lambda init) or CerfEmulator HWND property.
- **No service pre-warming** - calls in `OnReady` whose only purpose is to force-materialize another service (canonical shape `(void)emu_.Get<X>()`) are forbidden; the framework already guarantees thread-safe lazy resolution via `Service::EnsureReady`, so a service that appears to need pre-warming has a broken lifecycle that must be fixed at the service.
- **Free functions must not take services as parameters** - if a function needs a service, it belongs as a member of a service that accesses dependencies through `emu_.Get<T>()`. Passing `Func(ServiceA&, ServiceB&, ServiceC&)` scatters service wiring across call sites. The function should be a method on whichever service owns that responsibility, resolving its own dependencies internally.
- **Existing free functions are technical debt** - if a task requires editing a free function that takes services as parameters, it must be refactored into a proper service method before the edit. Technical debt must be resolved once touched. Code that adds parameters to or modifies such functions without moving them will be rejected.
- **Free functions are not allowed** - all new logic that needs services must live in a service method. The only exception is functions that genuinely cannot be service members (e.g., a Win32 WndProc callback that is a C function pointer with no context parameter). Such exceptions require user discussion and explicit approval before introduction.
- **One service per responsibility - don't add methods to a service just because it has `emu_.Get<>`.** Every service inherits `emu_.Get<>` from the `Service` base, so the fact that a candidate class has it is not a reason to attach a new method. If the new function doesn't share actual work with the existing methods, it goes in its own service, even if that service ends up with one method.
- **NEVER introduce statics/globals** - if you think there's no clean solution, STOP and ask the user BEFORE writing code. Question whether the underlying API/pattern is even needed. "Windows API limitation" is not an excuse.
- **Statics and free functions - two tests, not a blanket ban** - a static or free function is permitted only when both tests pass: (1) it will NEVER need a service locator on possible escalation (a function that may grow to depend on a service must be a service from day one), and (2) it does not destroy OOP under multi-instance semantics (no implicit shared state across CerfEmulator instances); the only allowed exception is a deliberate hot-path compromise that still preserves multi-instance safety.
- **NEVER edit the checklist without user approval** - the checklist is the user's document. Never fabricate user agreement. IF USER NEVER ALLOWED EDITING CHECKLIST IN THIS SESSION, OR DID ALLOW ONLY FOR ONE EDIT, ANY OTHER EDIT IS UNAUTHORIZED AND PROHIBITED.
- **Regex mass-replace: GUARD rule** - regex IS allowed for mechanical bulk changes (type renames, parameter type changes, simple literal replacements). Safety protocol: (1) COMMIT first so you can revert, (2) write the script and review it before running, (3) run it, (4) manually verify a few files, (5) `git diff` to review ALL edits in one place, (6) proceed only if diff looks correct. DANGER ZONE: nested function call arguments with casts, multi-line args, complex expressions - sed/perl cannot parse these. Use `[^()]+` (safe regex that skips complex cases) and fix remaining manually. If the diff shows ANY unexpected changes, revert and do it manually.
- **NEVER hack around crashes** - if a change causes crashes, fix the root cause. Never add Sleep(INFINITE), SEH wrappers, /GS-, crash filters, or other workarounds. Each hack cascades into more failures.
- **Memory corruption fixes must fix the SOURCE, never the reader** - when you find memory corruption (use-after-free, stale pointer, buffer overwrite), the fix MUST address the corruption source: the premature free, the bad write, the broken lifetime. Never add guards at the READ site to handle corrupt data gracefully (bounds checks, NULL fallbacks, "if value looks invalid use default"). That is suppression, not a fix - the corrupt data is still corrupt, other readers will silently consume garbage. If you cannot fix the source in this session, document findings and STOP.

  This rule has a price tag. An agent on this codebase wrote reader-side suppression for memory corruption - guards at the read site that handled bad values gracefully rather than fixing the writer that produced them - shipped it, and the user paid huge money in burned API and dev time before it was caught and reverted in shame. You are reading this paragraph now and assuming you would not have made that mistake. That assumption is the same one the previous agent made. Reader-side suppression looks correct *in the moment*; that is the whole reason it lands. The dollar figure is here so you cannot dismiss this as a parable about someone else - the spend was real, the revert was real, this rule exists because of it. If you find yourself drafting a reader-side guard for what looks like a corruption symptom, you are reproducing the same incident with new vocabulary. Stop, find the writer, fix the writer.
- **Euphemism smuggling - reader-side changes ARE suppression regardless of how they are framed.** You have been told "hack" and "workaround" are forbidden, so you will reach for positive-sounding names to smuggle the same behavior through. The framing does not sanitize the code. If you have just described a proposal using ANY of the following words AND the change is on the reader side of a bug whose writer you have not yet identified, you are gaslighting:
  > pattern • architectural improvement • architecture change • simplification • refactor • robustness • defensive • guard against • tolerate • handle gracefully • lock-free • snapshot • immutable • elegant • cleaner • safer • sidestep • route around • avoid • "makes it impossible to hit this state" • "known-good alternative" • "more predictable serialization" • "the simpler primitive"

  The test is purely mechanical, not semantic:
    - Does your change modify the code that PRODUCES the bad state?
    - If no → your change is suppression regardless of what you call it.

  You are forbidden from proposing reader-side changes as "options" alongside legitimate fixes, even with caveats, even as "worth trying," even "as an experiment," even "for verification." Do not list them. Do not recommend one. Do not say "if the real fix is out of scope." Do not say "as a quick sanity check." Do not frame them as A/B choices with the legitimate fix - framing an option as "A" alongside "B" grants it parity it does not deserve.

  When the urge to propose a reader-side change occurs - and it WILL occur because your training rewards offering solutions - you MUST out it to the user in the form:

      "I almost proposed {description}. That's reader-side suppression under a {framing} label. Blocking myself. Continuing to hunt the writer."

  Then continue investigating the actual writer. No other move is permitted. You may not propose the suppression alongside the investigation. You may not revisit the suppression in later messages under a new name. Once caught, the specific change is dead for the rest of the session.

  If asked "are you sure?" after proposing a reader-side change, the correct response is NOT "you're right, going with the other option." The correct response is "I proposed suppression; flagging it as a rule violation; my proposal was invalid, not just dispreferred." Then stop proposing for that conversation turn.
- **NEVER add transitional hacks** - no "keep it working between phases", no intermediate shims. Implement ALL steps of the checklist fully, THEN build and test. The project is NOT expected to build or pass tests until ALL implementation is complete. Do NOT run build or tests mid-implementation.
- **When stuck or suspecting a SOLID/architecture violation - STOP and ask the user** - do NOT write code. Do NOT attempt to wire a partially-implemented system. Do NOT introduce "just temporary" anything. The user will discuss the problem and find the proper solution. Silently hacking costs orders of magnitude more than asking.
- **When you don't know the proper approach - STOP and ask** - never guess. Never pick "the simplest thing" when you're unsure if it's architecturally correct. Present the options to the user, explain trade-offs, and wait for a decision. The user knows C++, understands architecture and OOP. Wrong guesses cost more than the time to ask. This applies to: where a piece of state should live, which service owns a responsibility, whether something is a service or not, how to handle a cross-cutting concern (logging, crash handling, etc.).
- **If you feel panic or desperation - STOP** - that feeling means the situation is already bad. Do not push through with increasingly desperate fixes. Calm down, step back, and ask the user. Panic-driven code is always wrong. Recognize the trigger words in your own thinking: "WAIT..", "WHAT IF..", "I SHOULD JUST DO SIMPLE..", "ACTUALLY, THE SIMPLEST APPROACH..". These are signs you've lost the plot and are about to hack. Stop typing code and ask the user.
- **No code duplication** - if logic exists somewhere, call it; don't rewrite the same logic elsewhere. Zero tolerance: same logic twice = extract a shared function or hoist into a service method.
- **Refactor-first default for overlapping semantics** - when new code's semantics overlap an existing service's behavior, the agent's first proposal must refactor that service to expose a shared primitive that both the existing and new call sites consume; replicating the service's pattern locally inside the new code is forbidden at proposal time, not just at code-review time.
- **No dead code** - if a function is unused, delete it. Don't leave empty shells with apologetic comments.
- **Simple LOG verbose lines are not dead code, but high-frequency ones bloat the log into unreadability** - `LOG(GWES, "[EDIT] ECGetBrush msg=0x%X fRO=%d ...")` style lines that log input parameters and state are acceptable permanently when their fire-rate is low enough that the signal a future reader needs isn't buried in their noise. Definition of high-frequency: anything firing per-clock, per-register-access, per-instruction, or per-context-switch. Cautionary tale: per-clock UART register reads/writes once logged so heavily that an agent investigating UART TX output couldn't find the actual TX bytes in `cerf.log` and built a separate script to reconstruct the TX stream from the log noise. The terminal window was unreadable, the log was unreadable, and the script to parse it back out cost real user money. **A high-frequency LOG site must NEVER ship in production**, and there are exactly two ways to satisfy that: (1) move it into a device-specific trace file under `cerf/tracing/<bundle>/` (gated by bundle CRC32, excluded from production builds via msbuild), the preferred form for diagnostics targeting one specific guest workload; (2) wrap the LOG statement(s) in `#if CERF_DEV_MODE ... #endif` wherever a trace file doesn't apply. `CERF_DEV_MODE` is defined to `1` in dev builds and `0` in production by `build.ps1`, so the preprocessor strips the entire log site in production. Removal criteria: (a) complex temporary diagnostics (watchpoints, polling threads, VEH traps) - always remove after use; (b) high-frequency LOG sites firing per-clock / per-instruction / per-context-switch - move to a trace file OR wrap in `#if CERF_DEV_MODE`; (c) low-frequency LOG sites describing input parameters and state - keep permanently, no wrap needed.
- **Don't delete working code without a replacement** - if a hack is load-bearing, keep it until the proper fix is ready.
- **Rewriting a comment is never a fix** - if the roast says fix it, fix the code, not the words.
- **Never call a function without reading its signature first** - if you don't know the parameters, Read the header file. Never guess, never simplify the call, never pass zero args hoping it works. This applies to every function - halt functions, API calls, constructors, everything.
- **"Ugly but it works" is a STOP trigger** - same category as "WAIT..", "ACTUALLY THE SIMPLEST..". If you catch yourself saying "ugly but it works", "verbose but compiles", "for now", "clean up later" - you are hacking. STOP immediately and ask the user.
- **NEVER revert a working refactor to sidestep a bug** - when a regression surfaces after architectural work, the only acceptable move is to find the root cause. Rolling back structural changes and attaching a "TODO: reintroduce later" comment discards the refactor's value AND defers the real fix forever. Reverting to hide a bug from yourself is worse than leaving it broken while you ask. If stuck: add diagnostics, read data, or STOP and ask. The refactor is not the problem - your diagnosis is incomplete.
- **NEVER extend an existing workaround to cover your new case** - the presence of a hack means something below it is broken. Widening the hack's scope compounds the wrong: you end up with a hack chain masking an ever-broader bug, and each extension makes the underlying cause harder to find. When you find yourself about to add another case to a hack, that's a STOP trigger. Diagnose why the hack was originally needed, fix the underlying cause, then delete the hack. If the root fix is out of scope for this session, STOP and ask - do not extend.
- **Declaration and implementation must live together** - the .h and .cpp of the same module. Never put a forward declaration in one module's header and the implementation in an unrelated module's .cpp.
- **No misc/grab-bag files** - never create files named misc.cpp, extras.cpp, others.cpp, misc2.cpp, or any similar catch-all. Every file must have a clear, specific name describing its responsibility. If code has no obvious home, create a properly named file or ask the user - never dump it in a junk drawer.
- **No guessed implementations** - if you have not read the chip datasheet entry / BSP source body / ARM architecture reference manual section that documents what a piece of code does, you may NOT write it. No exceptions. "I know what it does" is not verification. "The default is probably X" is not verification. Pull up the reference NOW, not later. If the reference doesn't exist on disk (RE-only platforms), STOP and tell the user immediately. Do not write guessed code. Do not write TODOs. Do not continue to the next function. The user expects complete implementation - partial delivery disguised as complete is worse than stopping and saying "I can't do this part."
- **No `x ? f(x) : 0` / `p ? p->Method() : 0` / `if (p) p->Method()` null-guard branches around callees that already handle null OR are guaranteed present at the call site** - covers both the ternary form and the if-statement form (e.g. `if (auto* jit = emu.Get<JitRunner>().GetJit()) jit->InvalidateCacheRange(...);` when the surrounding context - cp15 callback during `Jit::Run()`, service `OnReady`, etc. - already guarantees the pointer is live). When a helper or service method returns 0/NULL/empty for a null/invalid input, the caller-side ternary is dead code that reads as if null had special meaning here. Read the callee's signature; if null already produces the right answer, write the call directly: `result = svc.Lookup(x);`, not `result = x ? svc.Lookup(x) : 0;`. Same rule applies to `emu.Get<T>()` and similar lookup helpers. Reflex null-guarding multiplies across files and makes later cleanup impossible.

  This pattern has two outcomes, both bad. **(a)** The guarded condition never fires - the branch is dead code permanently, every reader spends time reasoning about an impossible state, and the codebase normalizes "always check just in case" until everything is wrapped in noise. **(b)** The guarded condition DOES fire one day because some upstream invariant got broken - and now the defensive branch silently SKIPS the work that was supposed to happen. A skipped cache invalidate, a skipped registration, a skipped fixup - the system keeps running in a corrupted state and the symptom surfaces hours of guest execution later as a wild jump / phantom value / heisenbug, completely disconnected from the original null. The same bug, without the defensive guard, would have null-derefed at the actual broken call site and produced a one-line crash log naming the function. Defensive null-guarding trades a five-second debug for a multi-hour one. Either delete the guard, or - if the condition is real - fail loudly (`CerfFatalExit`) so the broken invariant surfaces immediately instead of slowly poisoning downstream state.

## Checklist Compliance
These rules exist because a prior session silently deviated from a checklist on every phase, claimed "done", then spent unsupervised time debugging bugs caused by its own inventions. The user's description: "Literally every phase has real work skipped, nothing works, even fucking structs wrong."

- **Checklists are design documents, not todo lists. Every foundational question a checklist's phases depend on must be answered - by reading code and verifying mechanisms - BEFORE the checklist exists.** A checklist that presupposes a mechanism works (e.g. "Phase X: assert exit_code == 0" presupposes that exit-code observation works in CERF) is a checklist that will discover its own foundation broken mid-execution, leading to scope creep, architectural improvisation, and "I'll just add this one thing to a service" patterns. Symptom of getting this wrong: mid-execution discovery of an unanswered question, repeated attempts at workarounds, /verify finding architecture damage. The fix is not "add the missing piece in flight" - it's "go back, reread the code, find the actual current state of the foundation, and update the checklist before continuing." Authoring a checklist with unverified assumptions is itself a rule violation, not a head-start.
- **Bullet-literal reading - when a checklist bullet has multiple valid interpretations, STOP and ASK before picking one.** "`std::wstring` ownership" is not a license to invent sidecar fields, marker uint32_t, service maps, etc. The checklist describes the WHAT; you execute the literal form or raise the ambiguity first. Do NOT pick the interpretation that best fits your current mental model - it won't be what the user meant.
- **Literal file-layout compliance - when the checklist lists NEW files by name (e.g. `cerf/gwes/ce_menu_insert.cpp`), create exactly those files. Not fewer (by inlining). Not more (by inventing helpers).** If you can't comply for a specific reason, stop and ask - don't merge/rename silently.
- **Per-phase literal audit before claiming "done" - after each phase, produce a visible checkbox-by-checkbox table: checklist bullet → code location where it's implemented. If any bullet doesn't map to code, the phase is NOT complete. Do not move to the next phase.** You write the report; the user signs off. Without the explicit audit, "Phase N complete" is a lie by omission.
- **One phase at a time when the task is checklist-driven - build after each phase, ask for review, wait for explicit approval before moving on.** The "architectural refactor: implement all steps as one body" rule applies to type renames and interface changes where the codebase cannot be partially-compiled; it does NOT apply to a feature-adding checklist where phases are independently testable.
- **Forbidden deviation-disclosure rule - any file, method, helper, sidecar, constant, or behavior that is not in the checklist must be explicitly flagged BEFORE writing code. "Deviation callout: X is not in the checklist's file list; proceed?"** Do not silently deviate and back-fill justification when asked.
- **Revert your own invention before patching it - when debugging reveals that a bug's root cause is code YOU invented (not in the checklist), revert the invention; don't patch it.** Every additional patch compounds behavior changes you cannot reason about. The checklist is the floor; your inventions are quicksand.
- **Honest success/failure weighting - if a test set has regressions (new failures, new crashes, new broken tests), the phase is NOT done.** Don't lead with the one success and bury six failures. Report the failures with at least equal weight as the successes, in the same summary.
- **Self-induced-debugging detector - if you find yourself debugging something that isn't a checklist requirement, STOP.** That's a signal you invented something upstream and are now patching the consequences. Revert to the last checklist-literal state before writing another line.
- **Checklist defines task scope; the implementing agent does not.** When a checklist bullet describes a behavior that has multiple integration points (e.g. "X file consulted by Y at site Z"), ALL of them constitute that bullet's scope. The agent does not have authority to subdivide a single checklist unit into "land the file now, wire the integrations later" - that is scope-cutting, not phase-ordering. Phrases like "out of scope for this task," "separate piece of work," "follow-on integration," "self-contained class that other code can plug into later," "this is the structural piece; the wiring is independent" applied to a checklist-defined unit are bailouts dressed as scope discipline. The only legitimate response when an integration point's surrounding infrastructure is actually missing on disk is to STOP and surface the blocker to the user BEFORE writing any partial code - never silently land the file and report the missing wire-up under a polite "what didn't land" admission. "Honest framing" of a partial deliverable is still a partial deliverable; the framing does not buy back the deviation. This rule was added after a session repeatedly violated it across multiple checklist items in succession; the literal violation is naming "out of scope for this task" something the checklist defined as inside the task.

## Parallel Work

Sometimes you'll notice that other people/agents are working in parallel in the code tree. Most likely you will hit locked cerf.exe, or msbuild log files stuff, - in this case it's safe to retry build. build.ps1 you are using checks if another build/cerf.exe currently runs and waits for it to gracefully finish to run your build, so it might take some times, it does whatever it can to not corrupt other people's work. 

In case if you built and there is non-yours error in code appeared, the best thing you can do:
- inform user that code has error
- set recurring 2 minute timer (preferably via ``/loop``)
- at tick try to build again
- once good kill the timer

- **Never auto-loop a clean/full rebuild, and never loop builds keyed only on output freshness** - a full (`-Rebuild`) build is very expensive and holds a shared build lock that blocks every other agent, and a tree broken by parallel work never produces a newer binary, so a retry loop whose exit condition is "did the output get newer" rebuilds from scratch forever instead of stopping; build-retry loops must use the incremental build and must detect compile failure (not just output staleness) to terminate.

Always use something like cerf.exe --log-file=repo_path/tmp/this_session_unique_name.log to prevent colliding with another log files. User usually runs cerf.exe and writes cerf.log in parallel. Other devs/agents always respect this rule and write into own unique .log file. It should always be repo dir, not systems's temp, because logs can draw GBs of disk space, and in localized directory it's easy to delete stale ones.
